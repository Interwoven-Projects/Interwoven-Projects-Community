# Attribution & Credit Guidelines

**Fair, visible, and portable credit is one of the core values of Interwoven Projects.**

This document explains how we record contributions so that everyone who helps bring a project to life receives meaningful recognition.

## Why We Take Attribution Seriously

- It builds trust in a community of strangers and acquaintances.
- It creates portable reputation — your work here can be referenced elsewhere.
- It encourages high-quality contributions because people know their name and role will be attached.
- It honors the reality that great projects are almost never the work of one person, especially across disciplines.

## Core Principles

1. **Everyone who makes a meaningful contribution is credited.**
2. **Roles are descriptive, not hierarchical.** A "Domain Expert" on a physics project may have contributed as much or more than the person who first proposed the idea.
3. **Credit is recorded in a durable, human-readable place** (usually a `CONTRIBUTORS.md` or Contributors section in the project README).
4. **Credit should be easy to maintain** as projects evolve.

## Standard Contributor Roles

We use a shared vocabulary of roles that translate well across fields:

| Role                  | Description                                                                 | Examples |
|-----------------------|-----------------------------------------------------------------------------|----------|
| **Originator / Idea** | First person or group who proposed the core concept                         | "Came up with the initial project idea on X" |
| **Project Lead**      | Coordinates the project, keeps momentum, helps resolve blockers. **Must be an approved Moderator** who oversees the project and makes sure all ethical guidelines are being followed | Often the person who feels most ownership |
| **Domain Expert**     | Brings deep knowledge in a specific field the project needs                 | Physicist advising on quantum concepts; Historian providing accurate context |
| **Researcher**        | Gathers, synthesizes, or analyzes information, data, or sources             | Literature review, data collection, background reading |
| **Writer / Narrator** | Crafts prose, documentation, explanations, or final outputs                 | Blog post, paper section, project narrative |
| **Developer / Builder**| Implements software, models, simulations, tools, or technical systems     | Writing code, building prototypes, data pipelines |
| **Designer**          | Visual design, UX, systems thinking, diagrams, branding                     | Creating figures, interface mockups, information architecture |
| **Reviewer / Critic** | Provides thoughtful feedback that improves the work                         | Code review, manuscript review, reality-checking claims |
| **Facilitator**       | Helps the group stay organized, runs meetings, manages the project thread   | Scheduling, summarizing discussions, keeping the repo clean |
| **Data / Experiments**| Designs or runs experiments, collects empirical data, validates models      | Lab work, surveys, simulations, field observations |
| **Artist / Creative** | Contributes artistic, narrative, or experiential elements                   | Illustrations, sound design, interactive experiences |
| **Connector**         | Brings in new people or bridges between disciplines                         | "Introduced the philosopher to the coding team" |

**Notes on roles:**
- People often wear multiple hats on one project.
- Roles are assigned by mutual agreement, usually proposed by the Project Lead and confirmed with contributors.
- If a contribution doesn't fit neatly, we create a custom role or use a free-text description.

## How Credit Is Recorded

### For Active Projects
Every project that reaches the "Active" stage should have one of these:

1. **Preferred**: A `CONTRIBUTORS.md` file in the project folder (or linked repo) with a table.
2. A clearly marked **"Contributors & Credit"** section in the main project README.

Example table format (include Email where contributors are comfortable sharing it publicly):

```markdown
## Contributors & Credit

| Name / Handle          | Primary Role(s)              | Contributions                                      | X / Contact          | Email                     |
|------------------------|------------------------------|----------------------------------------------------|----------------------|---------------------------|
| Alex Rivera            | Originator, Project Lead     | Initial concept, coordination, final synthesis     | @arivera             | alex@example.com          |
| Dr. Priya Patel        | Domain Expert (Physics)      | Quantum mechanics accuracy, model review           | @priyaphysics        | priya@example.com         |
| Jordan Lee             | Developer + Writer           | Core simulation code, documentation                | @jordanlee           | jordan@example.com        |
| Sam Chen               | Designer                     | Diagrams, project visual identity                  | @samchen_design      | sam@example.com           |
```

### For Completed or Published Work
- All credited contributors should appear in the final artifact (paper, tool, website, exhibition, etc.) whenever possible.
- A link back to this GitHub project or the CONTRIBUTORS.md is strongly encouraged.

### Lightweight / Early Stage Projects
For very early discussions or one-off contributions, a simple list in the GitHub Issue or Discussion thread is acceptable. When the project becomes "Active," migrate the credit into the durable format above.

## Updating Credit Over Time

- Anyone can propose credit updates via a **Contribution Update** issue (see templates).
- Major changes should be discussed with the Project Lead and the people whose credit is being updated.
- Credit is never removed without very good reason (e.g., contributor requests it).

## Special Cases

**Large or anonymous contributions**
- Group contributions (e.g., "the Tuesday philosophy reading group") can be credited as a unit.
- Truly anonymous contributions are possible but we prefer named credit when feasible.

**AI / Tool-assisted work**
- If significant work was done with AI tools, we note the human guidance, curation, and final responsibility. The human(s) who directed and verified the work receive credit.

**Inherited or forked projects**
- Original contributors from outside this community should be credited where known.

## Questions or Disputes

If there is any uncertainty or disagreement about credit:
1. Start a respectful conversation in the project thread.
2. Tag the Project Lead and `@maintainers` (or equivalent).
3. If needed, escalate to the broader community via Discussions.
